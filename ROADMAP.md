# Roadmap

Phases, not dates. Each phase should produce something usable end-to-end before the next begins. Scope within phases is directional; items move as implementation teaches us.

> Canonical roadmap. Session handoff and working logs live in `claudedocs/` (untracked, maintainer-local).

## Phase 1 — MVP: SQL Server, CLI, the full loop (complete)

The core loop — `inspect` / `diff` / `apply` (+ history ledger) / versioned migrations / idempotent programmable-object routing — is implemented and verified end-to-end against LocalDB (see History).

- [x] **Failure-semantics ADR** — [ADR-0004](docs/adr/0004-failure-semantics-and-resume.md) (Accepted): convergent re-run, migration/redefine script+ledger in one transaction, transactional declarative publish, failure rows; cross-strategy ordering documented as a limitation. Crash-consistency decisions implemented and e2e-verified.
- [x] **Rollback / down-migration ADR** — [ADR-0005](docs/adr/0005-rollback-semantics.md) (Accepted): rollback = apply the previous git state (no dedicated verb); down migrations not supported — roll forward.
- [x] **Credential handling policy** — `SCHEMORPH_URL` environment variable preferred over `--url`; password material redacted at every output sink (console text, JSON messages, error envelope, persisted ledger failure detail). Further credential sources (files, vaults) deferred until real demand.
- [x] **Typed error envelope + exit-code enum** — `SchemorphError {kind, code, message, hint}` on stderr, `ExitCode` enum, code→kind mapping single-sourced in the core; contract documented in [docs/errors.md](docs/errors.md).
- [x] **CI + regression-test infrastructure** — build/unit-test CI; real-database integration tests (env-gated `SCHEMORPH_TEST_URL`, CI mssql service container — leaner than the originally sketched testcontainers); golden-corpus diff regression suite with self-bootstrapping baselines (`tests/Schemorph.IntegrationTests/Corpus/`). Corpus grows as edge cases surface.
- [x] **DacFx version policy** — exact pin, upgrades via dedicated PR gated on golden-corpus verification; documented in CONTRIBUTING ("DacFx version policy").
- [x] **Release engineering** — tag-triggered workflow (`release.yml`): `dotnet tool` package + win-x64 / linux-x64 / osx-arm64 self-contained single-file binaries attached straight to the GitHub Release (no workflow artifacts); NuGet push gated on the `NUGET_API_KEY` secret. First actual release = a human pushes a `v0.x.y` tag.

Exit criterion — *a real project manages its schema exclusively through Schemorph* — **met in mechanism** (2026-07-12): a real 41-table/22-view SSDT-managed MES/OMS schema completed the full loop (see History), and the two adoption blockers it surfaced (deploy-script parsing, brownfield phantom redefines) are fixed. Continuous dogfooding continues alongside Phase 2+.

## Phase 2 — Agent-native surface

- [x] Stabilize and version the JSON plan schema, aligned with Terraform/OpenTofu machine-readable-plan conventions — format 1.0 (`changes[]` with per-change action lists), 1.1 (`planHash`); [docs/plan-format.md](docs/plan-format.md)
- [ ] JSONL streaming for long-running commands — deferred until a long-running consumer exists (design would be speculative before the agent-usability harness)
- [x] Agent-first CLI conventions: JSON default on non-TTY stdout, self-describing `schema` subcommand (verbs / flags / output formats / exit codes as a JSON manifest)
- [ ] `next_actions` hints in responses — deferred with MCP response design (additive)
- [x] MCP server (`schemorph mcp`, stdio) exposing inspect / diff / apply / status as tools — safety model realized: read-only/plan-only tools by default, apply gated behind a **required plan fingerprint** (`expectedPlanHash` / CLI `--expect-plan`, error `plan_mismatch`), connection string confined to the server environment
- [x] Schema-as-context: current schema state and plans as MCP *resources* — `schemorph://schema` (live database as desired-state SQL), `schemorph://schema/{kind}/{name}` (one object), `schemorph://plan` (current plan incl. `planHash`, desired state from `SCHEMORPH_SCHEMA_DIR` in the server environment); inspect rendering split from its disk sink (`DesiredStateFile` / `InspectOperation`) so tools and resources share one rendering
- [x] Plan explanations ("this change forces a table rebuild because...") — format 1.2: `explanation` on every change (deterministic classification/checksum rationale, rebuild warnings with cost note), `sql` on redefines (the exact idempotent script) and on declarative changes via update-script attribution from DacFx's own per-object markers (conservative: unattributable → `null`, never a wrong attachment; degradation guarded by the exact-pin DacFx policy)
- [x] Safety lint — `SCHEMORPH1xx` warning band ([errors.md](docs/errors.md)): plan-side rules on plan messages (NOT NULL addition without default / table-rebuild cost / destructive change included) and migration-side static checks on pending migration files (`migrations.warnings` on status/apply: TRUNCATE, unfiltered UPDATE/DELETE, GRANT/REVOKE/DENY). Conservative by contract — AST-proven findings only, warnings never change the exit code
- [x] Agent-usability verification harness — `AgentSurfaceTests` (integration suite): the exit-criterion scenario (edit SQL → diff → review plan → stale-hash refusal → gated apply → converged status) driven over the real MCP stdio surface with the official MCP client, parsing only machine contracts. Runs wherever the integration tests run (env-gated, CI service container)
- [x] Agent Skill packaging — [`skills/schemorph/SKILL.md`](skills/schemorph/SKILL.md) (Agent Skills format: the safe change loop, machine-contract branching, migration immutability, MCP mode, brownfield adoption); shipped once the CLI/MCP surfaces stabilized (plan format 1.2, lint band, resources)
- [x] GitHub Actions recipe: post the plan as a PR comment — [docs/recipes/github-actions-plan-comment.md](docs/recipes/github-actions-plan-comment.md) (self-contained: base-branch schema rebuilt in a service container, PR diffed against it; marker-based comment upsert; optional lint-code policy gate)

Exit criterion: an AI coding agent completes a schema change end-to-end (edit SQL file → review plan → apply to dev database) without human-oriented output parsing. *(Demonstrated 2026-07-12 and fixed as a regression check — `AgentSurfaceTests` runs the scenario over the real MCP surface on every integration run.)*

## Phase 3 — Hardening & ecosystem

### Done

- [x] Drift detection — `status` verb + MCP `schemorph_status`: the plan a diff would produce now (drift), ledger summary (by kind, failures, last activity), pending migrations; exit 2 on pending work
- [x] Brownfield onboarding, part 1: existing databases and SSDT trees are consumed as-is — non-model files (SQLCMD deploy scripts, seed DML) are classified out with per-file warnings, and history-less programmable objects matching their live definitions **reconcile** (recorded, nothing executed) instead of phantom-redefining (ADR-0002 addendum)
- [x] Retype-triggered dependency invalidation for programmable objects — a column retype left a dependent view's text byte-identical, so the redefine checksum skipped it and the database kept describing the old type. The comparison now reports tables whose *existing* column definitions change, programmable objects carry the tables they read, and the redefine plan invalidates what depends on them transitively, in dependency order, and inside the fingerprint the apply gate checks. Column additions and removals are deliberately not invalidations, so the common additive change still redefines nothing

### Open — code

Two things are open, and they differ in kind. The first is a **staged response to the
first production consumer's feedback** — three findings from a real 31-object apply
against a live database, triaged 2026-07-20 and all accepted. The second is a single
item that cannot move without a human decision.

#### Production-gate hardening — staged

The ordering is by *harm*, not by size. The consumer drew a wrong conclusion from a
plan (S1), reviewed an artifact they had reassembled themselves (S2), and planned for
a worse failure mode than the tool actually has (S3). Correctness of the signal comes
before the artifact built on it, which comes before the prose describing it.

**S0 — verification that can be trusted** *(gate for S1–S3)*

Every stage below lands its evidence in the integration suite. Fixing the measuring
instrument is not overhead here; it is the precondition for believing any of it.

- [x] **Flaky integration suite** — classes failed intermittently under a full-suite run and passed in isolation. The load DacFx was buckling under turned out to be the suite's own: xUnit's default is one collection per class run in parallel, and every class here creates and drops a database on one shared server while several drive comparisons or spawn a child `schemorph mcp` process per test. Two full runs failed the same way from both ends — a plan whose `changes[].sql` was null (`GenerateScript` failing, correctly degraded to `SCHEMORPH002`) and a login timing out in the post-login phase after 14.6s inside `TestDatabase.Dispose`. That second failure is what named the cause: a starved server, not an unreliable engine. These tests were always serial-resource-bound; the runner was simply never told, so it rediscovered the constraint at random. `DisableTestParallelization` states it, and the suite is deterministic at 39/39. The cost is honest — 13m46s serial against 8m46s parallel-and-failing — and the standing DB-per-case proposal is where that time goes if it ever needs to come back

**S1 — the plan tells the truth about its own completeness**

- [x] **Restricted-comparison warning fired on the cause, not the effect** — `SCHEMORPH008` was keyed on DacFx reporting a missing `VIEW ANY DEFINITION`, so it fired on every least-privilege `db_owner` connection whether or not anything went unread. The first production consumer measured 64 objects live, 64 read, **0 missing** — and had to run a catalog query and a manual reconciliation to learn that, on every apply. The experiment settled the premise, and then went on to refute both fixes that had been proposed for it. Same database, same desired state, a structural change staged so completeness was observable, four logins: a `db_owner` without the server-scope grant read **everything**; a login with `DENY VIEW DEFINITION` came back **empty**; and a login **granted** `VIEW DEFINITION` at database scope but denied on the one changed object also came back **empty**. That last row is decisive — keying on the database-scoped permission, the fix this roadmap named, would have gone silent on an empty comparison, trading a false positive for a false negative in a safety warning. The other assumption also broke: DacFx does not omit silently. It reports an *error* in exactly the incomplete cases — *the reverse engineering operation cannot continue because you do not have View Definition permission on the '…' database* / *…because you have been denied View Definition permission on at least one object* — and nothing but the benign server-scope warning in the complete one. So the trigger is now that error, and the message echoes the engine's reason verbatim instead of asserting one, because "no permission on the database" and "denied on one object" need different fixes. That asserted cause was the misdirection itself: fixing the trigger and keeping the old text would have been half a fix. DacFx's warning is narrower than it reads, too — it restricts the comparison "to database scoped elements *if the source is a database*", and this provider's source is always a dacpac. Pinned by a four-login integration axis and `docs/errors.md` rewritten around the measurement

- [x] **A top-level restriction flag — built, then removed as unreachable** — the triage's second half asked for `comparisonIncomplete` beside `hasChanges`, on the reasoning that a caveat living only in `messages[]` cannot participate in a decision made from `hasChanges`, which is precisely the trap the consumer fell into. It was built as plan format 1.3 and then measured, and the measurement killed it: the flag can never be `true` in a rendered plan, because `DiffOperation` fails on the very errors that would set it. An unreadable target does not produce a plan at all — `diff` exits `1` with `compare_failed`, and `apply` refuses the same way. Shipping the field would have documented a gate that never fires, which is the false-negative shape this whole item exists to eliminate. Reverted to format 1.2. What the consumer needed was already there and had been all along; what was wrong was only *which* message wore the `SCHEMORPH008` label, and that is the fix above. Pinned now by a test asserting the real contract, so a future change to warn-and-succeed on an unreadable target fails loudly

*Exit:* met, and by a simpler route than planned. The tool does not emit a plan it
cannot stand behind: `diff` and `apply` both fail on an unreadable target, and the
false alarm on the complete least-privilege connection is gone. There is no partial
plan for automation to misread, so there is nothing left for a top-level flag to say.

**S2 — the plan becomes an artifact a person can sign** *(after S1)*

- [ ] **No reviewable script artifact for a human approval gate** — production consumers gate schema changes on a person reading the DDL, and the tool computes exactly that but never emits it: the first such consumer parsed `diff --format json` and reassembled the SQL themselves, which makes the reviewed text a *different artifact* from the executed one. `diff --format sql`, rendering the plan in execution order with the `planHash` in its header, closes it — and pairs the fingerprint with the paper a human actually signed. Note this is **not** blocked by the apply-path attribution item below: that constraint is about generating a script and publishing from one comparison, and `diff` does not publish. It already holds the update script and every redefine script. Review-only by design — a directly executable artifact invites `sqlcmd`, which skips the ledger, the redefine ordering, and the migration run-once contract

Not a hard dependency on S1 — the header could carry today's warning. A
*correctness-of-signal* one: a warning that fires on every least-privilege connection,
printed at the top of the document a reviewer is supposed to read carefully, teaches
them to skip it. Warning fatigue is worst exactly there.

*Exit:* a reviewer reads one artifact covering the whole plan, and the hash in its
header is the hash the apply refuses to deviate from.

**S3 — the apply says what it did and what it left**

- [x] **Failure semantics were not a user-facing contract** — a consumer about to apply 31 objects to production asked what the database looks like if the 17th fails, found no answer in `--help`, the manifest, or `docs/`, and reverse-engineered the ledger to a *wrong* conclusion: the per-object rows with success flags read like a progress log, but declarative rows are written in one batch after the publish commits, so their 8 column alters were all-or-nothing and had in fact fully rolled back. [`docs/failure-semantics.md`](docs/failure-semantics.md) now answers the question in the form it was asked, works the 31-change example through both branches, and gives the ledger its own section — a table of what a row means per kind, and the statement that only two of the three kinds are a progress log. ADR-0004 had all the facts; what it is not is a document an operator reads before approving a production change. Discoverable from the three places that consumer looked: `apply --help`, the `schema` manifest (`docs.failureSemantics`, manifest 1.1), and the README

- [ ] **The failure envelope drops what the core already computed** — `ApplyOperation.Outcome` carries `Stage`, `Applied` and `Plan`, and the CLI's JSON error path keeps none of them, so a failure cannot name the stage it failed in or the changes that did commit. A redefine failure is worse: it rethrows past the operation entirely and surfaces as a generic `execution` error, hiding committed declarative changes completely. Nothing new needs computing — this is rendering. The same shape appears verb-wide: `catch (Exception)` plus a hardcoded hint on `inspect_failed`, `compare_failed` and `apply_failed` alike, which is how an `inspect` failure came to blame "the connection string and output directory" when neither was the cause. Fix them together: the code and hint should follow the exception, not the verb

- [ ] **Every verb's failure hint is a hardcoded guess** — the same consumer hit an `inspect` failure whose hint named "the connection string and output directory"; neither was the cause. The filename in the message (`schemorph-inspect-<guid>.dacpac`, a prefix used at exactly one call site) pins the throw to `Extract` writing into `Path.GetTempPath()` — an internal artifact the user never asked for, surfacing as a path they cannot connect to any argument they passed. They reasonably concluded `--out` was not being created; it is, recursively, and `CoreLoopTests` already passes a non-existent subdirectory to prove it. So the filed fix would have been a no-op, and shipping it would have closed the issue without fixing anything. Two things follow. The verb-level one: `inspect`'s temp workspace is the tool's own responsibility — guard it and stop leaking the internal filename (the comparison session leaks the same way). The structural one, which is the same defect as the item above: `catch (Exception ex)` plus a fixed hint string appears on `inspect_failed`, `compare_failed`, and `apply_failed` alike, so the envelope work should widen to make the code and hint follow the exception rather than the verb. **Reproduce first** — the consumer reports that `mkdir -p` on `--out` fixed it, and no theory yet explains that, so the failure is not fully understood

Last because the behavior is already correct — only its self-description is missing.
S2 shrinks the harm further: the recommended path already hands the operator a
reviewed artifact before anything runs.

*Exit:* the question that started this ("what does the database look like if the 17th
of 31 objects fails?") is answered by a document the operator can find, a failed apply
names its stage and what it had already committed, and no failure hint points at
something the tool has not actually checked.

#### Awaiting a human decision

- [ ] **Apply-path script attribution** — `diff` computes the update script and attributes it per change; `apply` does not (its plan is built with `UpdateScript: null`), so the same operation says less at the moment it executes: no per-change `sql`, and `SCHEMORPH102` (rebuild cost) cannot fire on the path that actually rebuilds.

  **Attempted and reverted (2026-07-20) — this is an engine constraint, not an oversight.** Calling `GenerateScript` on the comparison result before `PublishChangesToDatabase` makes the publish fail intermittently with *"the target database was modified after schema comparison was completed"*; DacFx documents apply and generate-script as alternative actions on a comparison, not composable ones, and there is no per-difference script in the API. The three ways around it are all worse than the gap: executing the generated script ourselves means implementing the SQLCMD dialect it is written in and replacing DacFx's publish semantics (ADR-0001/0004 territory); a second comparison for the preview reintroduces the diff-apply race the same-snapshot invariant removed; and doubling the comparison cost on every apply to keep both is not worth what it buys.

  Scope of the actual harm, stated honestly: with `--expect-plan` the user already holds the reviewed plan — the one from `diff`, with its SQL and lint — and the fingerprint proves the apply matches it. The gap bites the *ungated* apply, which is the mode this tool least wants to encourage. **Needs a human decision (and an ADR) on whether apply should stop delegating execution to DacFx's publish**; not autonomous work.

  Worth re-examining after S2: once `diff --format sql` exists and the documented path
  is review-then-`--expect-plan`, the ungated apply this gap bites is even less
  load-bearing, and the decision may reasonably resolve as *won't do*.

### Demand-gated — waiting for a signal, not for time

Each needs an observation before it is worth building. None is scheduled.

- [ ] Baseline/onboarding flow, part 2: absorbing existing migration *history* from other tools (Flyway/Liquibase tables) — no consumer has asked
- [ ] Multi-environment configuration (dev/staging/prod targets, per-target safety policy) — a config-file surface is demand-driven (2026-07-12, human decision); the design should leave the door open to policy-as-code without pre-building it. Current signal: one consumer runs three environments but has not asked for per-environment *policy*
- [ ] Examples and comparison guides — end-to-end walkthrough, brownfield adoption walkthrough, and the README's competitive claims written out with sources. This is the content the ecosystem work is actually made of. A documentation **site** is deliberately *not* scheduled: at twelve files and ~10k words the docs have no navigation problem a generator would solve, and adding one would mean a Node toolchain this repo does not otherwise have (2026-07-20, human decision). Revisit if the doc set outgrows the README index — the evaluation is kept in `claudedocs/proposals/`

## Phase 4 — PostgreSQL provider

Still deliberately last, and the **engine choice is still deferred** ([ADR-0003](docs/adr/0003-postgres-as-second-provider.md)). What changed (2026-07-20): a first committed consumer supplied a **behavioral** requirement set — managed/non-superuser Postgres with no extension dependency, the three-strategy model at parity, expression-normalized diffs that re-run empty, plan/error/MCP contracts identical across providers, and transactional-DDL apply atomicity as a Postgres-specific opportunity — plus the acceptance scenarios it will judge adoption by. Behavioral requirements do not pick an engine; they become the axes the engine is chosen on.

The gate, restated (2026-07-20): **Phase 3's staged production-gate hardening, S0 through
S3.** Not its demand-gated items, and not the item awaiting a human decision — both may
never acquire what they are waiting for, and neither should hold a phase hostage. This
does push Phase 4 out, and that is the intended trade: a first production consumer
reporting from a live database outranks a second provider whose own consumer has a
working alternative and has stated no deadline.

Directional sequence (each stage gated on the previous):

- [ ] Requirements capture — record the consumer requirement set and the one design question it opens: whether apply **atomicity** may differ per provider and, if so, that it must be declared in the plan rather than left implicit (ADR-0004 territory). Decided at kickoff, not before.
- [ ] Evaluate engine options as they exist *then* (psqldef subprocess, parser-library binding, native catalog comparison), scored against those requirements — extension-freedom, control over apply atomicity, control over expression normalization, and single-file distribution all cut against some candidates
- [ ] ADR recording the choice
- [ ] Implement behind the provider boundary in vertical slices; revise the boundary where reality disagrees with it. Expression-normalized structural diff is the highest-risk slice, not the largest — false re-diffs are this tool category's chronic defect
- [ ] The test that matters: identical commands, file layout, plan format, and ledger semantics on both databases — run against a real managed-Postgres-shaped target, not only a permissive local container

Doable before that phase, and worth doing on their own merits (not gates):

- [x] Idempotent re-diff pinned — "apply, then diff again, and the plan is empty" is a provider-independent invariant, now asserted on the composed core loop across the expression-bearing shapes (`ConvergenceTests`: literal/function defaults, CHECK constraints, computed columns, filtered indexes, keys and references, programmable objects). It found a real one on first run: SQL Server stores `CHECK (… IN (…))` as an OR chain and DacFx compares expressions as text, so that shape **never converges** — the database is correct, the plan is not. Frozen as an explicit known-limitation test and documented in [docs/limitations.md](docs/limitations.md) with the `inspect` remedy; the root fix would mean reimplementing the engine's expression normalizer, so it is a proposal, not a plan. Concrete evidence that "control over expression normalization" belongs among the engine-evaluation axes below — a subprocess-wrapping engine would not have it
- [x] Provider resolution seam — the concrete provider was constructed at ten call sites across the CLI and MCP surfaces; it is now selected once (`ProviderSelection.Current`, the composition root) and handed out paired with its ledger store, so a provider can no longer be combined with another provider's ledger. Deliberately not a plugin registry, DI registration, or `--provider` flag: per ADR-0003 the selection *policy* stays unspecified until a second implementation exists to shape it. Side effect worth the name: exactly one CLI file now references the provider assembly at all

## Watching (deliberately not planned)

- **Declarative seed/reference data** (cf. Atlas v1.1 "Declarative Data Management") — in direct tension with ADR-0002's anchor ("data changes are events, not state"); adopting it would require an anchor-revising ADR. Revisit only on real user demand.

## Explicitly not planned

- GUI (structured output exists so others can build one)
- Hosted service (would be additive if ever pursued; see [Design Principles §5](docs/design-principles.md))
- MySQL/SQLite/other providers — plausible eventually, not before the two-provider boundary has proven itself

- **2026-07-13** — **Brownfield adoption-blocker run (cycles 53–56)**: the 2nd consumer (yesung, MES/OMS) surfaced four brownfield-adoption blockers against v0.2.0, triaged and fixed at the root. **Security principals out of the declarative model** ([ADR-0006](docs/adr/0006-security-principals-out-of-declarative-model.md)): a code-generated desired-state never emits users/logins/roles/permissions, so DacFx's `DropObjectsNotInSource` was proposing to DROP live principals for being "absent" — the silent destruction §4 forbids; they are now excluded from the compare (`ConfigureCompareOptions` / `ExcludeObjectTypes`), which also mooted the two lower-severity findings around principal-drop misclassification and DROP-USER script mis-attribution. **Column-order noise silenced** (`IgnoreColumnOrder`): a mid-table column add was being read as a full table rebuild purely because DacFx defaults to positional comparison — Schemorph diffs *state*, not ordinal, so an additive column now stays an in-place `ALTER ADD` (the row-holding-table rebuild cost that `SCHEMORPH102`/`101` warn about is only claimed when a genuine rebuild remains). **Restricted-comparison honesty** (`SCHEMORPH008`): when the connection lacks `VIEW ANY DEFINITION`, DacFx silently restricts the comparison and omits changes to unreadable object definitions — a partial result that must not read as "in sync"; that restriction is now surfaced as an explicit incompleteness warning (Schemorph's own redefine path already degrades safely, so the silent-omission risk was the DacFx declarative path alone). One finding — gating table *rebuilds* as destructive (`F1b`) — was investigated and **routed to a human decision** (Pending Human Decisions in HANDOFF): the apply-time gate classifies on `RawChange` alone with no rebuild-ness available, so it needs new apply-path script-attribution plumbing, and "all rebuilds destructive" over-gates lossless rebuilds while the §4-faithful "data-losing only" needs drop/narrow detection — a real trade-off, and F3 removed its urgency. Packed locally as **0.3.0** for consumer validation; **released as v0.3.0 on 2026-07-14** after that validation confirmed all four fixes (see above). Tests 84 core + 73 provider + 27 integration.
- **2026-07-13** — **2nd dogfooding + safety-lint blind-spot fix (cycles 48–52)**: the released v0.2.0 was dogfooded end-to-end through the **installed global tool** against the real 41-table yesung schema — every plan-format-1.2 surface (per-change `sql`/`explanation`, `planHash`, safety lint) confirmed in the field over both the CLI and the MCP `schemorph://plan` resource. It surfaced and fixed a safety-lint blind spot: `SCHEMORPH101` (a NOT NULL column added without a default) fired only on the in-place `ALTER ADD` shape, not when DacFx *rebuilds* the table (mid-column insertion) — where the row-copy `INSERT` omits the new column, so the identical "fails on a row-holding table" hazard is present and a CI lint-code policy gate would false-negative on the higher-risk path. `NotNullAddVisitor` now also proves the rebuild shape (a CREATE TABLE's NOT NULL-without-default column absent from the row-copy INSERT targeting that same table), leaving fresh-table creates and copied columns unflagged; double-pinned by an attributor unit test and a real-DacFx integration test (`CoreLoopTests`). Behavior now matches the always-general `errors.md` contract (the code had been under-delivering the documented promise). Competitive table re-verified as-of-2026-07 (Atlas/Bytebase/Flyway/Liquibase claims hold; Liquibase strengthened by Secure 5.2 MCP governance). The fix is unreleased — recommends a **0.2.1** patch. Tests 84 core + 58 provider + 24 integration.
